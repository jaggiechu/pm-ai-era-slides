What's Different from Old-School PM

Try Faster. Build Earlier. Overturn Yourself.

  • Mini prototypes, not 10-page PRDs. Get feasibility signal in hours, not weeks
  • Re-evaluate after every model release. Workarounds for old limitations become unnecessary complexity
  • Unlearn your priors. Yesterday's best practice can be today's tech debt. Throw away your own work

Example: Meta Ads creative team — shorter cycles, everyone (eng, design, DS, UXR, PM) runs small experiments off-roadmap

UCLA

Eval = Trial and Error

Prompt to yourself: if you were the boss reviewing your employee's work, what would your rubric be?

  • Rubric won't be right the first time. Run 100-150 samples, self-label, discover where dimensions conflict and categories blur
  • Iterate the rubric, not just the prompt. Fix the eval before fixing the model
  • Mental model: onboarding a junior teammate. Not a one-time doc — continuous, structured support and calibration

Be Technical: Know the Boundaries

Level 1: What works today vs. what doesn't
Level 2: What gets unlocked in 3-6 months — and why
→ Build where need meets capability

  • Papers: Abstract → Conclusion → main body only if curious
  • Best stuff isn't in papers. Follow X, HN, builder communities
  • Try first, ask second. Before asking a human, ask AI. Hands-on > hearsay
UCLA

Use AI Well: PRD for AI

The old artifact: PRD for engineers. The new one: PRD for AI.

  • Prompt → Context → Harness Engineering — single-prompt chat is not the ceiling
  • Agent System Design — Proposer + Red-team + Judge + Critic, or domain-split agents
  • Start with a concrete project — build end-to-end with AI. A baby tracker, a content assistant. Learning by doing
